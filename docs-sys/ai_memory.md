# AI Memory — persistent

> Long-term, high-level-abstraction memory. Small + clean SoC. `./` relative paths only.
> Tracks how the system fits together: which adapters / ports / domains associate. Folders by default; files only on special occasions.

## Layout (hexagonal)

- Published library is `./lib/` (vanilla, framework-free); `./src/` is the Preact dev playground only (per `vite.config.ts`). Build entry: `./lib/index.ts`.
- Kernel contracts (the shared integration surface every layer conforms to): `./lib/core/contracts.ts`. Pure, no DOM/framework/ambient reads.

## Associations (adapter / port / domain)

- **Domain core** — `./lib/core/types/` (codecs: `primitives.ts`, `semantic.ts`, `money_location.ts` [money/country/phone], `collections.ts` [array/object/any]). Pure type semantics implementing `TypeDefinition`. Numeric fidelity types (`decimal`, `int64`, money amount) are string-encoded — never rounded. Locale-aware `view()` for numerics/time/money/country; canonical stays locale-independent.
- **Projection system (Phase 2)** — `TypeDefinition.project?()` returns extended projections attached to `FieldValue.projections` (valid path only): time → `{local,utc,epoch}`, money → `{amount,currency,formatted}`, country → `{code,name}`, array → `{count,items}`. Pipeline wires it; widget shows a view+projection preview.
- **Application/orchestration** — `./lib/app/`: `pipeline.ts` (compile engine), `field.ts` (field lifecycle), `aggregate.ts` (virtual-form + cross-field dependencies), `inference.ts` (convention-over-config), `context.ts` (the ONE ambient-read boundary → explicit `Context`).
- **Registry port** — `./lib/registry/registry.ts` implements `Registry`; single resolution path for type/widget/format/serializer/validator.
- **i18n adapter** — `./lib/i18n/`: `i18n.ts` (flat-key resolver, fallback chain), `dictionary.ts` (default English; generic `*:error.<code>` covers engine codes not in any type's `codes`).
- **State port + adapters** — port `StatePort` (contracts). Adapters: `./lib/state/memory.ts` (built-in default) and `./lib/state/nanostores.ts` (optional plugin). Removable without changing canonical output. The field lifecycle (`./lib/app/field.ts`) already pushes every snapshot through the port (`create`/`set`), so any port wired via `createRuntime({state})` receives all field updates. The nanostores adapter additionally returns a `NanoStatePort` exposing the reactive atom per id via `store(id)` (read-only `ReadableAtom`, for native reactive consumption; writes stay the field's job). `<ui-field>` keeps a STABLE field id across attribute-driven rebuilds (explicit `id` attr → else reuse first-build id) so its atom is stable; reach it via `el.fieldId` + `runtime.state.store(id)`.
- **Widget adapter** — `./lib/widget/`: `widgets.ts` (built-in DOM renderers by `use`; host visibility is NOT managed here), `element.ts` (`<ui-field>` = the single primitive; resolves a widget by `use` with fallback to `input`; manages own visibility, `setDependencyState()` applies dependency effects without rebuilding the field), `form_element.ts` (`<ui-form>` = virtual-form + dependency host; carries a STABLE `name`/`.formName` — author `name` attr if given, else auto-generated `ui-form-N` reflected back onto the attribute so it stays `querySelector`-addressable; stamped into `du-form-change` detail for submit routing; identity only — does NOT namespace the combined canonical record). Presentational only.
- **External-library widget set (optional, opt-in)** — a self-contained widget bundle under `./lib/widget/` rendering a third-party custom-element library instead of native controls. Binds purely by DOM contract (creates the element tag, reflects state onto its props, translates its change event back through `WidgetProps.onInput`) — imports NO external code, so the core build stays dependency-free; the library is a runtime peer loaded + initialized by the page. Covers every value-producing control of the library, each on a distinct `use` key (never overrides the built-in defaults): boolean toggle, single/multi option group(s), dropdown, numeric slider, numeric stepper. Multi-select keys onto the `array` type by emitting a comma-separated raw (which `array.parse` accepts). DRY structure: a shared factory owns the host lifecycle + field chrome (label/help/preview/errors) + event→prop routing; each widget supplies only create/bind/reflect; shared helpers cover option-set building, change-key diffing, value↔string coercion, and numeric reflection. Registered on a runtime's registry via a `register…Widgets(registry)` helper exported from the composition root. The dev playground loads the sibling library via a `serve`-only path alias (`vite.config.ts`) + a local module-shim declaration; the published build never references it.
- **Dependency surface (cross-field)** — `./lib/app/expression.ts` (safe condition parser: `==,!=,>,<,>=,<=`, `and`/`or`, no eval), `./lib/app/dependency.ts` (`requiredIf`/`visibleIf`/`hiddenIf`/`validIf` → `Dependency`). `<ui-form>` reads `required-if`/`visible-if`/`hidden-if`/`valid-if` (+`message`) attributes on child fields. Aggregate exposes `issues()` + `effects()`; cycle detection ignores self-edges (valid-if reads its own target).
- **Composition root** — `./lib/index.ts`: `createRuntime()` wires registry+i18n+state+inference+context; `start()` also defines `<ui-field>`.
- **Author facade (singleton)** — `./lib/api/du.ts`: external convenience layer, the minimal entry point. `Du` singleton class + exported `du` instance. Lazily builds a runtime with the nanostores adapter as the DEFAULT state; `du.field(name, flatOptions)` and `du.form(name→options map, {dependencies})` return handles that auto-supply Context and expose the reactive atom (`handle.store`, `du.store(id)`). `du.define()` registers the custom elements (DOM-only, throws off-DOM). `du.configure({dictionary,locale,context})` before first use; `du.reset()` for tests. Composition only — changes NO canonical output; drop to `createRuntime()` for a custom state port.

## Notes

- Engine-level constraint codes (`required`,`choice`,`match`,`minItems`,`maxItems`,`unique`,`invalid`) are NOT in any type's `codes` array — they resolve via generic `*:error.<code>` dictionary keys.
- **Reactive filtering**: a known type filters proactively rather than erroring. `Field.filter()` delegates to `type.filter`; `WidgetProps.filter` lets the widget strip never-valid chars live (caret-preserving) so e.g. a letter in a number vanishes instead of showing "invalid".
- **Locale-driven numerics**: `./lib/core/types/numeric_locale.ts` (shared by primitives + money) reads the active locale's group/decimal glyphs from Intl. `filterNumeric(raw, allowFraction, constraints, ctx)` keeps digits/sign/decimal/group; `toCanonicalNumeric(raw, constraints, ctx)` strips group seps and maps the decimal → `.` before strict parse. So en `1,234.5` and de `1.234,5` both canonicalize to `1234.5`; canonical stays locale-independent. Fast-path `NUMERIC_FASTPATH` is permissive (rejects letters only).
- **`dotDecimal` flag** (`Constraints.dotDecimal`, `<ui-field dot-decimal>`): forces `.`=decimal / `,`=group regardless of locale (`numericSeparators()` returns this when set). View still locale-formatted; canonical unchanged. NOTE: this required adding `constraints` as the 2nd param to the `TypeDefinition.filter`/`parse` codec methods (contract change; pipeline + `Field.filter` pass it; all type files updated).
- **Phone E.164 charset**: `phone.filter` keeps ONLY a leading `+` and digits (ITU-T E.164); all formatting/letters stripped live.
- **Zoneless temporal types** (rfc.md §2.5 "zoneless temporal values", §7.3): `date` (canonical `YYYY-MM-DD`) and `time` (canonical `HH:MM:SS`) are wall-clock positions, NOT absolute instants — no timezone is attached, so canonical is timezone-independent (same input → same canonical under any `ctx.timeZone`). The §2.5 numeric-instant requirement is met by a derived projection only: date → `{iso,epoch@UTC-midnight}`, time → `{iso,msOfDay}`. Widgets: `date`→`<input type="date">`, `time`→`<input type="time">`, `calendar`(datetime)→`<input type="datetime-local">` — each emits exactly that type's canonical shape (distinct `defaultUse` per type; a wrong widget breaks the fast-path pattern). An absolute moment must be a `datetime`, or `time`+`timezone` combined downstream.
- Typecheck/build: `./tsconfig.app.json` includes both `src` and `lib`; `./tsconfig.lib.json` emits `.d.ts`. `npm run build` → `dist/index.mjs` + `index.iife.js` + types.
